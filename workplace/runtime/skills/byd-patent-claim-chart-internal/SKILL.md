# 比亚迪中文专利竞品比对（内网版）

## 目标

在**内网受限环境**下，围绕**比亚迪有权的中文专利**完成以下工作：

1. 锁定专利及应分析的全部权利要求
2. 拆解每条权利要求中的全部技术特征
3. 检索可能相关的竞品或车型
4. 仅基于允许访问的站点收集公开证据
5. 输出固定为 `.xlsx` 的逐条技术特征对比表

默认要求：**覆盖应分析的全部权利要求，并拆解每条权利要求中的全部技术特征。**

## 来源白名单

本 skill 只允许使用以下两个来源：

- **大为专利**
  用于：
  - 专利号
  - 专利名称
  - 权利要求全文
  - 专利法律状态/权属相关公开信息
- **汽车之家**
  用于：
  - 竞品车型信息
  - 参数配置
  - 车型页公开描述
  - 图片、配置表、公开页面中的技术线索

禁止：

- 引用其他网站的公开证据
- 使用“全网搜索后再筛”这种策略
- 在结果里放入非大为专利、非汽车之家的链接

如果某项特征无法仅凭这两个来源确认，直接写：

- `证据不足`
- `无法确认`

不要用猜测补齐证据缺口。

## 使用本 skill 时先读什么

这个 skill 的核心不是“写抓取脚本”，而是完成**比亚迪有权中文专利的 claim chart 初筛**。使用时优先读这些本地文件：

- 核心 workflow：`references/workflow.md`
- 站点接口与字段说明：`references/data-sources.md`
- 本地认证信息：`secrets.local.json`
- 固定 `.xlsx` 模板：`assets/byd_patent_claim_chart_template.xlsx`
- 大为专利最小取数脚本：`scripts/fetch_dawei_patent_claims.py`
- 汽车之家最小取数脚本：`scripts/fetch_autohome_series_params.py`

前 3 项决定“做什么”和“做到什么粒度”；后 2 项只是必要时的取数辅助，不要把脚本本身当成任务主体。

## 内网站点访问方案

本 skill **不要依赖 `web_search`**。内网方案应直接复用这两类访问方式：

### 1. 大为专利

优先使用大为专利的检索接口思路，而不是网页全文搜索。具体接口、字段和认证要求见 `references/data-sources.md`。如需直接抓权利要求，或先按主题词筛候选专利，都运行 `scripts/fetch_dawei_patent_claims.py`。但关键词检索必须被硬性限制为**比亚迪中文有权专利**；输出只是原始材料，后续仍需人工/agent 做权利要求拆解。

认证信息不要写进 `SKILL.md` 正文。优先从当前 skill 目录下的 `secrets.local.json` 读取；如果该文件不存在，再从环境变量读取：

- `DAWEISOFT_DEVICE_ID`
- `DAWEISOFT_TOKEN`

如果 `secrets.local.json` 存在，agent 和脚本应优先复用其中的大为凭证。

核心访问方式：

- 接口：`POST https://pat.daweisoft.com/api/innojoy-search/api/v1/patent/search`
- 常用请求头：
  - `token`
  - `deviceId`
  - `X-Group-Env: PAT`
  - `Origin: https://pat.daweisoft.com`
  - `Referer: https://pat.daweisoft.com/searchresult`
- 若内网环境要求代理，则复用用户环境中的代理设置；不要自己假设外网搜索可用

两种典型用法：

1. **按关键词或主题找候选专利**
   - `searchType = patent_list`
   - `query = (<关键词>) AND CAS=(比亚迪) AND CC=(CN)`
   - 返回后仍需再次过滤，只保留 `CC=CN` 且 `CAS` 命中比亚迪的记录
2. **按公开号精确拉权利要求**
   - `searchType = baseinfo_detail`
   - `query = PNM=<专利公开号>`
   - `fields` 至少包含：`PNM,TI,CL,CLM,CLMN`

关键点：

- 要拿权利要求全文，`searchType` 必须是 `baseinfo_detail`
- 中文权利要求优先取 `CL.CLZH`
- 如果只有 XML 片段，要清洗标签后再使用

### 2. 汽车之家

优先使用汽车之家的站内搜索、车型页和参数接口，不依赖外部搜索引擎。具体接口、字段和推荐提取项见 `references/data-sources.md`。如需抓汽车之家证据，不要把关键词写死在脚本里；应先由 agent 基于专利标题和全部权利要求自行生成关键词，再运行 `scripts/fetch_autohome_series_params.py --keyword ...`。这个脚本会先做汽车之家站内搜索，再自动补抓命中候选车系的车型页和参数页，还会对搜索结果里的文章/论坛链接再抓一层正文页，提取可直接写进 claim chart 的证据摘录。脚本输出只是证据素材，不能替代技术特征映射。

核心访问方式：

- 站内搜索意图接口：
  - `GET https://sou.autohome.com.cn/afu_search_proxy_api/qp_full`
- 站内搜索结果接口：
  - `GET https://sou.api.autohome.com.cn/v1/search`
- 新能源月销榜：
  - `GET https://www.autohome.com.cn/web-main/car/rank/getList`
- 车型页：
  - `GET https://www.autohome.com.cn/{series_id}`
- 参数接口：
  - `GET https://www.autohome.com.cn/web-main/car/param/getParamConf`

建议流程：

1. agent 先根据专利内容生成检索关键词
2. 先用站内搜索接口查文章、视频、车系相关结果
3. 从搜索结果中提取候选车系、候选文章链接和摘要
4. 对候选文章/论坛链接再抓一层详情页，提取正文、作者、发布时间和按段落去重后的证据摘录
5. 再打开命中的车系页，从 `__NEXT_DATA__` 中提取代表车型信息（例如 `hotSpecId`）
6. 再调用参数接口，并围绕显式关键词检索该车系的公开字段
7. 最后把可支撑专利技术特征的文章证据、参数证据、车型页链接写入比对表

适合优先抽取的汽车之家字段包括：

- 能源类型
- 电池类型
- 电池能量
- 电池能量密度
- 续航相关参数
- 快充/慢充相关参数
- 车系名 / 品牌名 / 代表车型名

如果汽车之家上没有明确公开某项特征，就写 `证据不足`，不要用车型常识替代证据。

## 脚本调用方式

agent 在这个 skill 下如果要调用脚本，优先直接按下面的方式执行，不要自己重新发明命令格式。

### 1. 搜索比亚迪中文有权专利

如果用户先给的是技术主题，而不是专利号，先这样调用：

```bash
python3 workplace/runtime/skills/byd-patent-claim-chart-internal/scripts/fetch_dawei_patent_claims.py \
  --keyword "电池" \
  --page-size 10 \
  --output workplace/output/<session_id>/byd_battery_patents.json
```

这个模式下，脚本会自动强制加入：

- `CAS=(比亚迪)`
- `CC=(CN)`

并在结果中再次过滤，只保留“比亚迪中文有权专利”。

### 2. 拉大为专利权利要求

按公开号获取专利基础信息和中文权利要求时，优先这样调用：

```bash
python3 workplace/runtime/skills/byd-patent-claim-chart-internal/scripts/fetch_dawei_patent_claims.py \
  --pnm CN203964454U \
  --output workplace/output/<session_id>/cn203964454u.json
```

如果有 `secrets.local.json`，脚本会优先读取其中的大为凭证；没有再退回环境变量。

### 3. 搜汽车之家并补抓车系参数

agent 必须先根据专利标题和全部权利要求自己生成关键词，然后再调用脚本。默认调用方式：

```bash
python3 workplace/runtime/skills/byd-patent-claim-chart-internal/scripts/fetch_autohome_series_params.py \
  --keyword "热泵空调 小米SU7" \
  --keyword "PTC 小米SU7" \
  --max-results 8 \
  --max-series 3 \
  --output workplace/output/<session_id>/xiaomi_su7_autohome.json
```

这个脚本会做四件事：

1. 用关键词调用汽车之家站内搜索
2. 返回文章/视频/车系候选结果
3. 对文章/论坛结果再抓一层详情页，提取正文摘要和按段落去重后的证据摘录
4. 自动对命中的候选车系补抓车型页和参数页

如果用户已经明确给了某个 `series_id`，可以显式传入，缩小补抓范围：

```bash
python3 workplace/runtime/skills/byd-patent-claim-chart-internal/scripts/fetch_autohome_series_params.py \
  --keyword "热泵空调 小米SU7" \
  --series-id 6962 \
  --max-results 8 \
  --max-series 1 \
  --output workplace/output/<session_id>/xiaomi_su7_autohome.json
```

如果只是调试，想看原始搜索返回、原始参数分组，再额外加：

```bash
--include-raw
```

### 4. 输出文件约定

- 脚本输出 JSON 时，统一写到 `workplace/output/<session_id>/`
- 文件名尽量和专利号、竞品名相关，便于后续生成 `.xlsx`
- 不要把最终 claim chart 直接混在中间取数 JSON 里
- 汽车之家搜索结果里应优先使用 `results[].detail.content_preview` 和 `results[].detail.evidence_snippets`
- `evidence_snippets` 默认是按段落匹配关键词、按内容去重后的候选证据摘录，优先直接写入 claim chart

## 重要边界

- **不要直接下“构成侵权”“一定侵权”“法律上成立”之类结论**
- **不要只分析权利要求1，除非用户明确要求只看某几项**
- 只做：
  - 大为专利上的专利信息读取
  - 汽车之家上的竞品公开信息提取
  - 技术特征拆解
  - 证据化对比
  - 风险初筛
- 所有竞品特征都必须有**公开证据**
- 合理推断必须单独标注为“推断”，不能伪装成明确证据
- 只处理**比亚迪有权的中文专利**；如果专利权属不清、不是中文专利、或不是比亚迪相关专利，先提示用户补充或确认

## 输入预期

优先使用以下任一输入：

- 专利号
- 专利全文或权利要求文本
- 专利名称 + 专利号
- 明确指定的竞品范围
- 明确指定的车型范围

如果用户没有给全，优先补齐以下最关键要素：

1. 专利号或完整权利要求
2. 要分析的权利要求范围
3. 竞品或车型范围

如果用户没有明确限定权利要求范围，默认处理全部权利要求，包括：

- 全部独立权利要求
- 全部从属权利要求

## 核心流程

### 1. 先确认分析对象

先确认：

- 专利是否为**比亚迪有权的中文专利**
- 当前关注的是哪些权利要求；如果用户未限定，默认是全部权利要求
- 是做单一竞品比对，还是做候选车型筛查

专利文本、权利要求、法律状态、公开文本优先从大为专利接口读取。  
如果用户给的是技术主题而不是专利号，先用大为接口检索候选专利，但候选范围必须被强制限定为**比亚迪中文有权专利**。

### 2. 拆解权利要求

默认从独立权利要求开始，但必须继续覆盖全部从属权利要求，除非用户明确缩小范围。

拆解原则：

- 按最小可对比技术单元拆分
- 每条特征尽量保持“一个明确技术点”
- 不要把多个技术点糊成一句
- 一条权利要求里如果包含并列条件、范围条件、结构条件、位置条件、功能条件、公式条件，要继续拆开，直到每行只对应一个可核验技术点
- 给每条特征编号，例如：
  - `1.1`
  - `1.2`
  - `1.3`

强制要求：

- **每一条权利要求都要拆**
- **每一条权利要求中的每一个技术特征都要拆**
- 不允许只保留某条权利要求的摘要结论
- 不允许只挑一部分特征做对比
- 如果某条从属权利要求是在上位权利要求基础上的进一步限定，也要把新增限定特征逐项拆出并单独比对

### 3. 在白名单站点内设计检索路径

检索只允许围绕两个站点展开：

- 大为专利：
  - 查专利全文
  - 查权利要求
  - 查专利名称
  - 查权属或法律状态相关公开项
- 汽车之家：
  - 查月销榜接口
  - 查车型页
  - 查参数配置接口
  - 查公开介绍文案
  - 查车型对比页

不要把外部行业资料、拆解文章、新闻站作为补充来源。

执行时优先级如下：

1. 如果已知专利号，大为接口按公开号精确拉专利和权利要求
2. 如果只给主题词，先用大为接口检索候选的比亚迪中文有权专利
3. agent 先根据专利内容生成关键词
4. 汽车之家脚本先按这些关键词做站内搜索，拿文章/视频/车系候选结果
5. 再对命中的候选车系补抓车型页和参数页
6. 只在必要时阅读对应网页正文补摘要

### 4. 候选竞品筛查

如果用户没有先给出竞品名单，先根据专利技术主题在汽车之家里筛出候选车型或产品方向。

筛查时重点看：

- 产品类别是否相同或相近
- 汽车之家页面上是否有足够公开参数
- 是否存在可回链的页面证据

如果汽车之家上公开资料明显不足，不要硬比，直接标记为“不适合做公开证据比对”。

### 5. 收集证据

每一条竞品特征都要附证据，证据至少记录：

- 证据标题
- 证据链接
- 来源类型
- 证据摘录或摘要
- 访问日期

来源类型只能写成：

- `大为专利`
- `汽车之家`

如果一条技术特征需要多个证据支撑，也只能来自这两个来源。

### 6. 做技术特征映射

对每一条权利要求技术特征，逐项写出：

- 权利要求技术特征
- 竞品对应特征
- 证据链接
- 证据摘要
- 匹配判断
- 不确定点

匹配判断统一使用以下口径之一：

- `明确匹配`
- `部分匹配`
- `可能匹配`
- `证据不足`
- `明显不匹配`

如果某条权利要求包含多个技术特征，必须拆成多行逐项判断；不要整条权利要求只给一个总判断。

### 7. 输出最终结果

输出主表至少包含以下列：

| 权利要求 | 特征编号 | 权利要求技术特征 | 竞品/型号 | 竞品对应特征 | 证据摘要 | 证据链接 | 匹配判断 | 备注 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |

最终交付格式固定为 `.xlsx`。

优先参考模板文件：

- `assets/byd_patent_claim_chart_template.xlsx`

当用户要求输出文件时：

1. 先参考模板结构，确认工作表、顶部信息区、统计区、权利要求标题行、技术特征明细行的布局
2. 再由 agent 直接编写本次任务专用 Python 脚本，生成最终 `.xlsx`
3. 最终文件必须写入当前会话目录 `output/<session_id>/`

## 输出写法要求

### 表格要求

- 一条技术特征一行
- 同一条权利要求下的全部技术特征必须完整展开
- 一个竞品一个独立表或在表中明确标识型号
- 证据链接只能是大为专利或汽车之家
- 备注列专门写争议点、缺口、推断点

### 模板要求

模板主表必须支持：

- 一件专利下的多条权利要求连续展开
- 每条权利要求前有一条“权利要求标题行”
- 每条权利要求下的多项技术特征连续逐行展开
- 不能把一整条权利要求压缩成单行总表述

### 文字总结要求

表格所在 `.xlsx` 里应保留这些核心模块，允许样式更简洁，但模块不能少：

- 报告标题
- 专利/竞品基本信息
- 分析口径与边界
- 结论统计区
- 逐条权利要求技术特征对比主表

可在工作表顶部摘要区简短写明：

- 哪些特征证据较强
- 哪些特征仅部分覆盖
- 哪些特征证据不足
- 建议进一步人工核查的点

不要直接写成法律结论。

## 质量检查

交付前至少检查：

1. 是否明确限定为比亚迪有权中文专利
2. 是否只使用了大为专利和汽车之家
3. 是否覆盖了应分析的全部权利要求
4. 是否把每条权利要求的全部技术特征拆到了可逐项比对的粒度
5. 是否每条竞品特征都附了公开链接
6. 是否所有链接都来自白名单站点
7. 是否区分了明确证据和推断
8. 是否避免直接下侵权法律结论
9. 如果生成了文件，文件是否写到当前会话目录并可下载

## 实施建议

在具体任务里，agent 应按 workflow 驱动，而不是按脚本驱动：

1. 先用 `references/workflow.md` 确认分析对象、输出口径和证据边界
2. 必要时用 `scripts/fetch_dawei_patent_claims.py` 获取权利要求原文
3. 由 agent 根据专利内容生成一组汽车之家检索关键词
4. 必要时用 `scripts/fetch_autohome_series_params.py --keyword ...` 先做站内搜索，再补抓车型公开参数
5. 由 agent 拆解全部权利要求的全部技术特征，并逐项匹配竞品证据
6. 最后基于 `assets/byd_patent_claim_chart_template.xlsx` 生成最终 `.xlsx`

不要尝试恢复通用 `web_search` 工作流；这个内网 skill 就按站点接口驱动执行。
